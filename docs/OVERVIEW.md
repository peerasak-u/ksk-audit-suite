# KSK Audit Suite — Overview

## เป้าหมาย

สร้าง Claude skill สำหรับสำนักงานสอบบัญชี เพื่อช่วย generate เอกสารกระดาษทำการ (working papers) จากเอกสารดิบที่ได้รับจากลูกค้า โดยอ้างอิงจากตัวอย่างจริง 16 เคสในโฟลเดอร์นี้:

- `4_ตัวอย่างไฟล์ลูกค้า/` — เอกสารดิบที่ได้รับจากลูกค้า (input)
- `5_ตัวอย่างไฟล์ผลลัพธ์/` — ผลลัพธ์ที่สำนักงานสร้างจริง (target output) — ใช้เป็น ground truth

โฟลเดอร์ลูกค้าทั้งสองฝั่งตั้งชื่อเหมือนกันทุกตัวอักษร (1:1 pairing) รูปแบบ: `<S/WS/งาน#> [<รหัสลูกค้า>] <ชื่อเล่นพนักงาน> <ชื่อบริษัท บจ./หจก.> <DDMMYY วันสิ้นงวด>`

เป้าหมายด้าน consistency ที่ผู้ใช้ระบุ: **ไม่ต้องการให้เนื้อหาเหมือนเดิมทุกครั้ง แต่ต้องการให้ "ฟอร์แมต" เหมือนเดิมทุกครั้ง** — ไฟล์ที่ generate ออกมาต้องคงโครงสร้าง/schema/หน้าตาเดียวกันเสมอ แม้เนื้อหาข้างในจะต่างกันตามข้อเท็จจริงของลูกค้าแต่ละราย

## ไฟล์เป้าหมาย 5 ไฟล์ ต่อลูกค้า 1 ราย (ในโฟลเดอร์ 5, ใต้ `WP/`)

1. `1 Planning ....xlsx` — แผนการตรวจสอบ
2. `2 หน้ารายงาน....docx` — หน้ารายงานของผู้สอบบัญชี (audit report)
3. `3 CIT50 ....pdf` — แบบภาษีเงินได้นิติบุคคล (ภ.ง.ด.50)
4. `4 งบการเงิน....xlsx/.xls` — กระดาษทำการงบการเงิน (workpapers)
5. `ใบปะหน้างบการเงิน.docx` — ใบปะหน้า

## สิ่งที่พบจากการเปิดไฟล์จริง (cell/formula/field level)

### โฟลเดอร์ 4 (input จากลูกค้า) — ไม่มีโครงสร้างคงที่
- อยู่ใต้ subfolder ชื่อประมาณ `เอกสารประกอบงบการเงิน` แต่การจัดระเบียบต่างกันมากในแต่ละเคส: บางเคสไฟล์แบน ([103]: 6 ไฟล์), บางเคสซ้อนลึกเป็นหมวด ([312]: 33 ไฟล์ แยกโฟลเดอร์ `2.แบบภาษี ใบเสร็จ/`, `3.หนังสือรับรอง และ บอจ.5/` ฯลฯ)
- ประเภทไฟล์หลากหลาย: PDF (แบบภาษีสแกน, statement ธนาคาร, บอจ.5), Excel (GL/TB/asset schedule จากลูกค้า), `.rar` (เอกสารรวมมัด), `.jpg` (รูปถ่ายเอกสาร), แม้แต่ `.md`/`.txt` (memo อิสระ)
- ชื่อไฟล์ไม่มี schema ตายตัว มีคำผิด รูปแบบวันที่ไม่คงที่

### `ใบปะหน้างบการเงิน.docx`
- เนื้อหาแค่ 3 บรรทัด: ชื่อบริษัท, หัวเรื่องเอกสาร, วันที่สิ้นงวด
- Templatable ~100%

### `2 หน้ารายงาน.docx`
- เทียบ 2 เคสจริง ([103] vs [202]): **ทุกย่อหน้าตัวอักษรเหมือนกันทุกตัวอักษร**
- จุดที่ต่างกัน: ชื่อบริษัท (ซ้ำ ~4 จุด), วันที่งบ, ชื่อ/เลขทะเบียนผู้สอบบัญชี, เมือง, วันที่ลงนาม
- บางเคส ([233]) มีย่อหน้าเสริม เช่น กรณีผู้สอบบัญชีปีก่อนเป็นคนละคน
- นิติบุคคลประเภท หจก. ใช้ฟอร์แมตคนละแบบ ("รายงานการตรวจสอบและรับรองบัญชี" แทน "รายงานของผู้สอบบัญชีรับอนุญาต")
- Templatable สูงมาก — base template + optional block ตาม flag (เช่น `has_prior_auditor_note`, `entity_type`)

### `1 Planning.xlsx`
- 14 sheet เหมือนกันทุกเคสที่เช็ค: `ข้อมูลลูกค้า, ประเภทบัญชี, 000, 001, 101, 102, 103, 202, 203 TB (2), 301, 302, 401, 601, 608`
- sheet `ข้อมูลลูกค้า` เป็น key-value form ล้วน (col A = label ตายตัว ~30 field, col B = ค่าที่กรอก) เช่น ชื่อ, ที่อยู่, เลขผู้เสียภาษี, กรรมการ, ผู้สอบบัญชี/ทีม, วันที่ต่างๆ, ค่าธรรมเนียม
- **พบว่า sheet อื่นอีก 9/14 sheet ดึงค่าจาก `ข้อมูลลูกค้า` ผ่านสูตรอยู่แล้ว** (ตัวอย่าง: sheet `202` มี 48 สูตร อ้างอิงกลับ 45 จุด, sheet `601` มี 18/18, sheet `102`/`103`/`301`/`302`/`401`/`608` ก็อ้างอิงเกือบทั้งหมด) → งานจริงของ skill นี้คือกรอกแค่ sheet เดียว ที่เหลือ cascade อัตโนมัติผ่านสูตรที่มีอยู่แล้วในไฟล์ต้นฉบับ
- **ข้อยกเว้น**: sheet `301` (การประเมินความเสี่ยง) และ `203 TB (2)` (วิเคราะห์การเปลี่ยนแปลง TB) มี narrative จริงที่เขียนขึ้นจากการอ่านเอกสารลูกค้า เช่น "รายได้จากการขายลดลง เนื่องจากสินค้าต้องมีการแก้ไข อย." — ต้องอ่านข้อมูลจริงมาเขียน ไม่ได้เป็น fill-in-the-blank แต่ sheet `301` มี legend คงที่ (Assertion code: Completeness=C, Accuracy=A, Existence&Occurrence=E, Right&Obligation=R, Cut-off=CO, Valuation=V, Presentation=P) ที่ใช้บังคับ vocab ได้

### `3 CIT50.pdf`
- ตรวจสอบแล้วเป็น **AcroForm จริงของกรมสรรพากร มี 925 field ชื่อตายตัว** (เช่น `nDate1`, `NID.0`-`NID.3`) — เปิดด้วย `pypdf`
- กรอกด้วยการเขียน field ตรงๆ ได้ deterministic เต็มรูปแบบ (ไม่ต้องสร้าง PDF ใหม่)
- ความยากอยู่ที่ตัวเลขก่อนกรอก (การปรับปรุงกำไรทางบัญชี → ทางภาษี) ไม่ใช่กลไกการกรอกฟอร์ม
- พบว่าบางเคส ([431], [448]) ในไฟล์งบการเงินมี sheet ชื่อ `ภาษีเงินได้`/`คำนวณภาษี` อยู่แล้ว → การคำนวณภาษีเกิดขึ้นใน workpaper งบการเงินอยู่แล้ว ไม่ใช่ทำใน skill CIT50 — skill CIT50 แค่ดึงตัวเลขจาก cell ที่รู้พิกัดมายัดใน field mapping (ทำ mapping ครั้งเดียว reuse ได้)

### `4 งบการเงิน.xlsx`
- เทียบ sheet name ของ 5 เคสจริง: **ไม่เหมือนกันเลย** (8-18 sheet ต่อเคส) ขึ้นกับว่าลูกค้ารายนั้นมีบัญชีอะไรบ้าง เช่น `wp_เงินสด`, `wp_เงินให้กู้`, `ทะเบียนคุมที่ดิน`, `สัญญาเงินกู้`, `เจ้าหนี้อื่น`
- sheet ที่ recur บ่อย: `0000` (ปก/index), `งบการเงิน`/`งบการเงิน68` (งบจริง), `TB_68`/`TB68`/`TB2025` (trial balance), `wp_เงินสด`, `wpเจ้าหนี้การค้าและเจ้าหนี้อื่น`
- นี่คือ workpaper จริง ไม่ใช่ template — เป็นไฟล์ที่ซับซ้อนที่สุดและ automate ยากที่สุด

## สรุปตาราง templatability

| ไฟล์ | Format แน่นอน/ตายตัว? | เนื้อหาต้องใช้วิจารณญาณส่วนไหน |
|---|---|---|
| ใบปะหน้างบการเงิน (docx) | ใช่ 100% | แทบไม่มี |
| หน้ารายงาน (docx) | ใช่ (มี optional block) | เลือก opinion type / entity type / ย่อหน้าเสริม |
| Planning (xlsx) `ข้อมูลลูกค้า` | ใช่ — เขียน sheet เดียว ที่เหลือ cascade ผ่านสูตรเดิม | ไม่มี (เป็นข้อมูลบริษัทล้วน) |
| Planning (xlsx) `301`, `203 TB` | โครงสร้าง table ตายตัว แต่เนื้อหาไม่ตายตัว | วิเคราะห์ความเสี่ยง/TB จากเอกสารจริง |
| CIT50 (pdf) | ใช่ — AcroForm 925 field ตายตัว | ตัวเลขปรับปรุงภาษี (มาจากไฟล์งบการเงิน) |
| งบการเงิน (xlsx) | ไม่ — จำนวน/ชื่อ sheet ต่างกันทุกเคส | ทั้งหมด เป็นงาน audit จริง |

## แนวทางสถาปัตยกรรมที่วางไว้ (ยังไม่ implement)

หลักการหลัก: **แยก "การสกัด/ร่างเนื้อหา" (งานของ LLM) ออกจาก "การเขียนไฟล์" (งานของสคริปต์ deterministic)**
- LLM มีหน้าที่อ่านเอกสารดิบ → สกัดเป็น JSON/dict ตาม schema คงที่ (หรือร่าง narrative ตาม vocab ที่กำหนดไว้ล่วงหน้า)
- สคริปต์ python (openpyxl / docxtpl / pypdf) เป็นตัวเขียนไฟล์จริงจาก locked template — ให้ input data เดิม จะได้ format และโครงสร้างเดิมทุกครั้ง

แผนสกิลคร่าวๆ:
1. **Ingest/scaffold skill** — อ่านโฟลเดอร์ลูกค้า (แตก `.rar`, จำแนกประเภทไฟล์) แล้วสรุปว่ามีเอกสารอะไรครบ/ขาดก่อนเข้าสู่ขั้น generate
2. **ใบปะหน้า + หน้ารายงาน** — docxtpl mail-merge, deterministic เต็มรูปแบบ
3. **Planning** — เขียนเฉพาะ sheet `ข้อมูลลูกค้า` ด้วย openpyxl (ห้ามแตะ sheet อื่นเพื่อไม่ให้สูตร/format เดิมเสีย) ส่วน `301`/`203 TB` แยกเป็นงาน narrative ที่ต้อง ground กับเอกสารจริงเสมอ
4. **CIT50** — ทำ field-name mapping ของ 925 field ครั้งเดียว แล้วดึงตัวเลขจาก sheet `ภาษีเงินได้`/`คำนวณภาษี` ในไฟล์งบการเงินมากรอก
5. **งบการเงิน** — ยากสุด ต้องเป็น pipeline ผู้ช่วยจัด TB/สร้าง supporting schedule โดยอิงโครงสร้างจากเคสใกล้เคียง ต้องมี human review เสมอ

## เครื่องมือที่ต้องใช้ (เช็คในเครื่องแล้ว)

- `openpyxl` (3.1.5) — มีอยู่แล้ว, อ่าน/เขียน xlsx พร้อมสูตร/format ได้
- `python-docx` — มีอยู่แล้ว, อ่าน/เขียน docx ได้
- `pypdf` — ต้องติดตั้งเพิ่ม (`pip3 install pypdf`), ใช้กรอก AcroForm ของ CIT50
- `docxtpl` — ยังไม่เช็ค แนะนำสำหรับทำ mail-merge แบบ Jinja2 บน docx
- `unrar`/`patoolib` — ยังไม่มีในเครื่อง ต้องติดตั้งเพื่อแตกไฟล์ `.rar` ที่ลูกค้าบางรายส่งมา

# อัพเดตการออกแบบ (2026-07-14)

## Database กลาง 3 ไฟล์ = master lookup ไม่ใช่แค่ context เสริม

พบว่าไฟล์ที่ขึ้นต้นด้วย `Database` ที่ root เป็น **แหล่งข้อมูลหลักของ metadata ลูกค้า** และเป็นพื้นที่ที่มนุษย์กับ agent ทำงานร่วมกัน (มีคอลัมน์สถานะการยืนยัน) — เอกสารดิบในโฟลเดอร์ 4 เป็น **แหล่งรอง** ใช้สำหรับตัวเลขงบ/รายชื่อกรรมการเป็นหลัก

### `Database งาน.xlsx` — ตารางงานหลัก (PK = เลขงาน)
- sheet `รวมรายชื่องบการเงิน` (398 แถว) คีย์ = `ลำดับงาน` **ตรงกับเลข `[NNN]` ในชื่อโฟลเดอร์ทั้งฝั่ง 4/5 เป๊ะ** → join ได้ 15/16 เคส
- field ที่ได้ทันทีจากการ lookup: `ชื่อกิจการ, เลขนิติ (tax id), รอบปีบัญชี, ที่มา, ชื่อผู้สอบ (ชื่อจริงผู้สอบ), ขนาดกิจการ(รายได้รวม), ดอกเบี้ยรับ, รายได้หักดอกเบี้ย, ค่าสอบจากงบการเงิน, เรียกเก็บจริง, จ่ายผู้สอบ, รูปแบบหน้ารายงาน (= opinion type), วันที่หน้ารายงาน (= sign date)`
- sheet `5ฟิลด์ ตรวจแก้` (คีย์ = ผู้สอบ + เลขที่งาน) มี `รูปแบบหน้ารายงาน, วันที่หน้ารายงาน, หมายเหตุ/แก้ไข` + คอลัมน์สถานะ `ยืนยันข้อมูลแล้ว / แก้ไขข้อมูลแล้ว` ← **จุด human-in-the-loop: agent เติม คนยืนยัน**
- sheet `ลูกค้า <ชื่อผู้สอบ>` (per-auditor) มี `ทุนจดทะเบียน, รายได้, ค่าสอบบัญชี, วันที่แจ้งสภา`

### `Database ข้อมูลผู้สอบ.csv.xlsx` — ทะเบียนผู้สอบ (7 คน)
- join ด้วย `ชื่อผู้สอบ` จาก Database งาน (ชื่อผู้สอบบัญชีแต่ละคนในสำนักงาน) → ได้ `เลข CPA TA, CPA/TA, คำนำหน้า, ชื่อ-นามสกุลเต็ม, สังกัด, เลข สนง, ที่อยู่`
- คอลัมน์ `CPA/TA` คุมฟอร์แมตหน้ารายงาน: CPA → "รายงานของผู้สอบบัญชีรับอนุญาต", TA → "รายงานการตรวจสอบและรับรองบัญชี"

### `Database เรทคิดเงิน Audit.csv` — ตารางเรทค่าสอบ (43 แถว)
- lookup `รายได้ (bracket) → ค่าสอบ` แยกตาม `ที่มา` (ชื่อผู้แนะนำลูกค้า ฯลฯ) เป็นการคำนวณ deterministic ของ `audit_fee`

## ข้อควรระวัง: ต้อง validate เลขงาน ห้ามเชื่อ DB ตาบอด
- เคส `[448]` โฟลเดอร์ = ชื่อบริษัทหนึ่ง แต่ job 448 ใน DB = ชื่อบริษัทอีกรายหนึ่งไปเลย (คนละบริษัท คนละชื่อผู้แนะนำ) → **ไม่ตรง** ต้อง flag
- เคส `[364]` **หายจาก DB** → ต้อง fallback ไปสกัดจากเอกสารดิบ
- กติกา: ingest ต้อง cross-check `ชื่อกิจการ` จาก DB เทียบชื่อในโฟลเดอร์/เอกสาร ถ้าไม่ตรงให้ตั้งสถานะ `⚠ ต้องตรวจมือ` ไม่ generate ต่อโดยอัตโนมัติ

## การตัดสินใจด้านสถาปัตยกรรม (ล็อกแล้ว)

1. **6 skill แยกอิสระ** ผูกกันผ่านเอกสารกลาง `CONTEXT.md` (1 ไฟล์ต่อ 1 ลูกค้า วางใน `5_.../<ลูกค้า>/CONTEXT.md`)
2. **ลำดับ build**: `audit-ingest` (ingest + scaffold) ก่อน → cover-report → planning → cit50 → financials (เฟสหลัง) → orchestrate (optional)
3. **งบการเงิน (skill 5) เลื่อนเป็นเฟสหลัง** ทำ 1–4 ให้จบก่อน
4. `audit-ingest` **ไม่เขียนเนื้อไฟล์ผลลัพธ์** — แค่ (ก) สร้างโครงโฟลเดอร์ `WP/` พร้อมช่อง 5 ไฟล์ และ (ข) เขียน `CONTEXT.md` พักไว้ให้ skill ปลายน้ำมาเติมทีละใบ

## รายการ skill (6 ตัว) แบ่งตามประเภทงาน

| # | skill | ประเภทงาน | หน้าที่ | trust level |
|---|---|---|---|---|

## `CONTEXT.md` schema — ทุก field มี provenance

หัวใจ: ทุกค่าต้องระบุ **source** (DB ไหน / เอกสารไหน / derived) และ **status** (`✔` ยืนยันแล้ว, `⚠` ต้องเติม/ต้องตรวจ) เพื่อให้คนรู้ว่าเชื่อได้แค่ไหน และให้สคริปต์ปลายน้ำ parse ส่วนบน (Client Profile) เป็น data ได้

โครง 4 ส่วน:
1. **Client Profile** — ตาราง `field | value | source | status` (มาจาก DB-join เป็นหลัก)
2. **Validation** — ผลการ cross-check เลขงาน vs ชื่อบริษัท, flag เคส [448]/[364]-style
3. **Document Inventory** — ผล scan/แตก folder 4 (มี/ขาดเอกสารอะไร + path)
4. **Judgement Notes** — human: opinion_type, has_prior_auditor_note, ข้อสังเกตอิสระ

field หลักใน Client Profile: `company_name, company_name_en, entity_type (บจ./หจก.), tax_id, address, directors[], registered_capital, revenue, period_end, prior_period_end, auditor_name, auditor_license, auditor_type (CPA/TA), audit_team, city, sign_date, opinion_type, audit_fee`
- ส่วนใหญ่ = DB lookup; `directors[]`, `address` (บางเคส), ตัวเลขงบ = ต้องสกัดจากเอกสาร (บอจ.5 / งบ)
